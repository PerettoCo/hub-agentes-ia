SOFIA · Rate Limit

Workflow SOFIA SDR que integra Quepasa → Chatwoot fazia requisições em rajada, estourando o rate limit da API do Chatwoot (erro 429). Um nó Wait com 500ms entre requisições eliminou os bloqueios.

Explicação Simplificada

O que aconteceu?

O SOFIA é um robô que recebe mensagens do WhatsApp (via Quepasa) e joga tudo no Chatwoot (sistema de atendimento). Ele processava várias mensagens de uma vez, fazendo muitas requisições ao Chatwoot ao mesmo tempo.

O Chatwoot tem um segurança na porta que diz: "calma lá, só aceito X requisições por minuto". O SOFIA ignorava isso e fazia tudo de uma vez, levando um "chega pra lá" (erro 429 — Too Many Requests). Cada execução tomava de 3 a 5 bloqueios.

É como tentar entrar 20 pessoas de uma vez numa porta giratória — a porta trava e ninguém passa.

O que mudou?

Agora o SOFIA espera 500 milissegundos (meio segundo) entre cada requisição. Em vez de mandar tudo em rajada, ele envia uma mensagem, espera meio segundo, envia a próxima, espera, e assim por diante.

É como organizar uma fila indiana: cada um espera sua vez. Meio segundo é tão rápido que o usuário final nem percebe, mas é o suficiente para o Chatwoot não travar.

Resultado esperado

Zero erros 429. Todas as mensagens do WhatsApp são registradas no Chatwoot sem bloqueio. O processamento fica um pouco mais lento (meio segundo por mensagem) mas 100% confiável.

Detalhamento Técnico

Problema Original

Workflow: [SOFIA] SDR IA — Quepasa → Chatwoot

Nó com falha: HTTP Request (Chatwoot API — POST /api/v1/conversations)

Erro: 429 — Too Many Requests | Retry-After: 60

Causa raiz: O workflow consumia mensagens do webhook do Quepasa e as enviava ao Chatwoot em paralelo (ou sequencial rápido demais). A API do Chatwoot tem rate limiting de ~60 requisições por minuto (configuração do plano). O n8n enviava todas as mensagens do lote (batch) em menos de 2 segundos, excedendo o limite e recebendo 429.

Impacto: 3 a 5 erros 429 por execução. As mensagens bloqueadas eram perdidas (sem retry automático) e precisavam ser reenviadas manualmente.

Solução Aplicada

Inserir nó Wait (Timer) entre cada iteração do loop, configurado com 500ms de intervalo:

Configuração do nó Wait:
  "mode": "duration"
  "duration": 500
  "unit": "milliseconds"
  "includeInput": true

Posicionamento no fluxo:

  • Trigger (Quepasa Webhook) → SplitInBatches (1 item por vez)
  • Wait (500ms) → aguarda meio segundo
  • Nó Code (formata payload Chatwoot)
  • HTTP Request → Chatwoot API
  • Loop → próximo item

Arquivo de correção: sofia-wait-rate-limit.json (1 nó adicionado)

VersionId: correcao-sofia-v1

Antes vs Depois

Antes:

  • Todas as requests em rajada (<2s)
  • 3-5 erros 429 por execução
  • Mensagens perdidas sem retry
  • Necessidade de reenvio manual

Depois:

  • 500ms de intervalo entre requests
  • Zero erros 429
  • 100% das mensagens entregues
  • Throughput: ~2 mensagens/segundo

Mitigação de Falhas

  • Rate limit reduzido no futuro → ajustar duration no nó Wait (ex: 1000ms)
  • Mensagens urgentes com atraso → não há prioridade, aplicar fila prioritária separada
  • Chatwoot retorna 429 mesmo com wait → implementar retry com backoff exponencial + nó Code
  • Lote muito grande (>100 msgs) → SplitInBatches garante espaçamento de 500ms entre cada uma

Plano de Rollback

Desabilitar ou remover o nó Wait entre o SplitInBatches e o HTTP Request. O rate limiting original volta a ocorrer.
