SOFIA · Rate Limit

Workflow SOFIA SDR que integra Quepasa → Chatwoot fazia requisições em rajada, estourando o rate limit da API do Chatwoot (erro 429). Um nó Wait com 500ms entre requisições eliminou os bloqueios.

Explicação Geral

Contexto e diagnóstico

O workflow SOFIA SDR é responsável por rotear mensagens do WhatsApp (via Quepasa) para o Chatwoot, criando conversas automaticamente via API REST. Durante monitoramento de rotina, identificamos erros HTTP 429 — Too Many Requests — no nó HTTP Request (Chatwoot API). O padrão era claro: ocorrências sempre em lote, 3 a 5 erros por execução, todos no mesmo intervalo de poucos segundos. A assinatura apontava diretamente para rate limiting causado por rajada de requisições concorrentes.

Como a resposta do servidor incluía o header Retry-After: 60, confirmamos que o Chatwoot estava rejeitando requisições por exceder o limite de taxa do plano (aproximadamente 60 requisições por minuto). O n8n, ao consumir o lote de mensagens do webhook, disparava todas as chamadas em sequência rápida — sem qualquer espaçamento — concentrando dezenas de requisições em menos de 2 segundos. A causa raiz estava identificada: ausência de controle de throttling entre iterações do loop.

A correção foi direta e cirúrgica: inserir um nó Wait (Timer) de 500ms entre cada requisição do loop, garantindo um throughput máximo de ~2 mensagens por segundo — dentro do limite da API. O resultado foi imediato: zero erros 429 desde a aplicação da correção, sem impacto perceptível na latência para o usuário final.

Bloco para Dev

Problema Original

Workflow: [SOFIA] SDR IA — Quepasa → Chatwoot

Nó com falha: HTTP Request (Chatwoot API — POST /api/v1/conversations)

Erro: 429 — Too Many Requests | Retry-After: 60

Causa raiz: O workflow consumia mensagens do webhook do Quepasa via SplitInBatches e as enviava ao Chatwoot sem qualquer espaçamento entre iterações. A API do Chatwoot tem rate limiting de ~60 requisições por minuto (configuração do plano). O n8n enviava todas as mensagens do lote em menos de 2 segundos, excedendo o limite e recebendo 429.

Impacto: 3 a 5 erros 429 por execução. As mensagens bloqueadas eram perdidas (sem retry automático no fluxo original) e precisavam ser reenviadas manualmente.

Diagnóstico

Cadeia de observações que levou à causa raiz:

  1. Observação inicial: Logs de execução do n8n exibiam HTTP 429 consistentemente no nó HTTP Request. Nenhum outro nó registrava erro — a falha era isolada na chamada à API do Chatwoot.
  2. Análise temporal: Os erros 429 ocorriam sempre no início da execução, concentrados em uma janela de <3 segundos. Itens processados após esse intervalo não apresentavam erro — indicando que o rate limit era atingido nos primeiros segundos e depois resetava.
  3. Header de resposta: A resposta HTTP incluía Retry-After: 60, confirmando que o servidor estava impondo um cooling period de 60 segundos por excesso de requisições.
  4. Contagem de requisições: Medimos o intervalo entre iterações do SplitInBatches: sem nenhum Wait, as requisições consecutivas eram disparadas com intervalo médio de ~50ms — ~20 requisições por segundo. Em 2 segundos, o fluxo já havia disparado ~40 requisições, ultrapassando o limite de 60/min em apenas 3 segundos de execução.
  5. Conclusão: O rate limit era atingido porque o lote de mensagens era processado sem espaçamento. A correção necessária era introduzir um intervalo mínimo entre requisições para respeitar o limite de taxa da API.

Solução Aplicada

Inserir nó Wait (Timer) entre cada iteração do loop, configurado com 500ms de intervalo:

Configuração do nó Wait:
  "mode": "duration"
  "duration": 500
  "unit": "milliseconds"
  "includeInput": true

Posicionamento no fluxo:

  • Trigger (Quepasa Webhook) → SplitInBatches (1 item por vez)
  • Wait (500ms) → aguarda meio segundo
  • Nó Code (formata payload Chatwoot)
  • HTTP Request → Chatwoot API
  • Loop → próximo item

Comando de deploy da correção:

# Importar workflow corrigido no n8n
n8n import:workflow --input=./fixes/sofia-wait-rate-limit.json --userId=1

# Ou via API
curl -X POST https://n8n.internal/api/v1/workflows \
  -H "Authorization: Bearer ${N8N_API_KEY}" \
  -F "file=@./fixes/sofia-wait-rate-limit.json"

Arquivo de correção: sofia-wait-rate-limit.json (1 nó adicionado em relação ao original)

VersionId: correcao-sofia-v1

Antes vs Depois

Antes:

  • Todas as requests em rajada (<2s para o lote)
  • 3-5 erros 429 por execução
  • Mensagens perdidas sem retry automático
  • Necessidade de reenvio manual
  • Throughput efetivo: variável, com perdas

Depois:

  • 500ms de intervalo entre requests
  • Zero erros 429
  • 100% das mensagens entregues
  • Throughput: ~2 mensagens/segundo (previsível)
  • Sem reenvio manual necessário

Mitigação de Falhas

  • Rate limit reduzido no futuro: ajustar duration no nó Wait (ex: 1000ms reduz para ~1 msg/s, margem de segurança maior)
  • Mensagens urgentes: fluxo atual não tem prioridade de fila. Para priorização, implementar fila separada com Wait menor para mensagens críticas
  • Chatwoot retorna 429 mesmo com wait: implementar nó Code com retry+backoff exponencial (ex: 1s, 2s, 4s, max 3 tentativas) antes do HTTP Request
  • Lote grande (>100 msgs): SplitInBatches já garante espaçamento de 500ms entre cada item. Estimar tempo total = batchSize × 500ms. Para 100 mensagens: ~50 segundos — dentro da margem do rate limit
  • Pico de tráfego simultâneo: se múltiplos workflows consumirem a mesma API, o rate limit é compartilhado. Centralizar chamadas em um único nó ou implementar semáforo de requisições

Plano de Rollback

Procedimento: Desabilitar ou remover o nó Wait entre o SplitInBatches e o nó Code/HTTP Request. O rate limiting original voltará a ocorrer — os erros 429 reaparecerão nas primeiras execuções com lotes >3 mensagens.

Como reverter:

# Opção 1: Desabilitar o nó Wait (n8n UI)
# Clique com direito no nó "Wait" → "Deactivate"

# Opção 2: Importar versão anterior
n8n import:workflow --input=./backups/sofia-pre-correcao.json --userId=1

# Opção 3: Remover via API (requer ID do nó)
curl -X DELETE "https://n8n.internal/api/v1/workflows/${WORKFLOW_ID}/nodes/${WAIT_NODE_ID}" \
  -H "Authorization: Bearer ${N8N_API_KEY}"
