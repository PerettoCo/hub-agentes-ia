Explicação Geral

Contexto do Problema

O SDR IA da GENICS depende de Redis como camada de cache para manter sessões, filas e dados temporários de prospecção. Essa instância estava hospedada na Redis Cloud (AWS ElastiCache) e acessada via DNS público. Em condições normais de rede, a conexão era funcional — mas o resolvedor de DNS do servidor começou a falhar intermitentemente, impedindo a resolução do hostname do Redis.

Identificamos imediatamente o padrão: o driver node-redis relata getaddrinfo ENOTFOUND quando não consegue traduzir o nome DNS para IP. A falha não era no Redis em si — era na camada de descoberta. A cada requisição de cache, o cliente Redis esperava até 17 minutos pelo timeout de resolução DNS, bloqueando toda a pipeline de prospecção. A decisão foi direta: remover a dependência de DNS removendo o Redis da nuvem.

A migração para Redis local (Docker no mesmo host do n8n) eliminou a resolução DNS da equação. Conexão via loopback, latência abaixo de 1ms, zero timeouts. A correção foi aplicada em produção sem interrupção adicional ao fluxo — o cache local passou a responder instantaneamente e o backpressure foi totalmente zerado.

Bloco para Dev

Problema Original

Workflow: [GENICS] SDR IA

Nó com falha: Redis (node-redis client)

Erro: getaddrinfo ENOTFOUND redis-xxxx.cache.amazonaws.com

Causa raiz: Redis Cloud (AWS ElastiCache) acessado via DNS público. O resolvedor de DNS do servidor falhou intermitentemente — sem resposta do nameserver externo, o driver node-redis não conseguia resolver o hostname, acionava timeout em ~17min (padrão TCP sysctl) e acumulava retentativas exponenciais.

Impacto: Toda execução do SDR IA que dependia de cache Redis ficava bloqueada durante a resolução DNS. O backpressure se propagava para os nós upstream e parava o fluxo de prospecção.

Diagnóstico

A investigação seguiu uma cadeia lógica de eliminação:

  • 1. Verificação do erro no n8n: Logs do n8n mostravam getaddrinfo ENOTFOUND repetido. O erro não era de autenticação ou conexão recusada — o hostname simplesmente não resolvia.
  • 2. Teste de resolução DNS no servidor: dig redis-xxxx.cache.amazonaws.com retornava status: SERVFAIL em rajadas intermitentes. nslookup confirmava falha parcial do nameserver configurado (/etc/resolv.conf).
  • 3. Confirmação de que o Redis Cloud estava online: Teste via outro host com DNS diferente provou que o endpoint Redis respondia normalmente — o problema era exclusivamente do resolvedor local.
  • 4. Decisão: Corrigir o resolvedor DNS do servidor seria tratar o sintoma. A causa estrutural era a dependência de rede externa para um cache que deveria ser local. Optou-se pela migração para Redis containerizado no próprio host, eliminando DNS da equação.

Solução Aplicada

Remover dependência externa de Redis. Substituir Redis Cloud por instância Redis local Docker no mesmo host do n8n:

docker run -d \
  --name redis-sdr \
  --restart unless-stopped \
  -p 127.0.0.1:6379:6379 \
  redis:7-alpine \
  redis-server --save 60 100 --loglevel warning

Ajustes complementares no servidor:

  • vm.overcommit_memory = 1 no sysctl.conf — evita que o Redis seja morto pelo kernel em sobrecarga de memória
  • net.core.somaxconn = 1024 — aumenta backlog de conexões
  • disable_thp = echo never > /sys/kernel/mm/transparent_hugepage/enabled — desabilita THP que causa latência esporádica no Redis
  • Healthcheck no container: redis-cli ping a cada 10s

Configuração no n8n: Alterar variável de ambiente REDIS_URL de redis://redis-xxxx.cache.amazonaws.com:6379 para redis://127.0.0.1:6379. Nenhuma alteração de código foi necessária — apenas a string de conexão.

Arquivo de correção: otimizacao-servidor-nos.md (tuning de servidor)

Antes vs Depois

Antes (Redis Cloud):

  • Tempo de conexão: 200ms-17min (intermitente)
  • Dependência de DNS externo
  • Custo mensal de ~US$ 15/mês
  • Backpressure travava o fluxo

Depois (Redis local):

  • Tempo de conexão: <1ms
  • Zero dependência externa
  • Custo: 0 (mesmo servidor)
  • Backpressure zero

Mitigação de Falhas

  • Redis Cloud DNS falha → Redis local não precisa de DNS, conexão é direta via loopback
  • Servidor reinicia → Docker restart: unless-stopped + healthcheck automático
  • Container Redis morre → Logs no journald + healthcheck reconecta em <10s
  • Falta de memória → vm.overcommit_memory=1 evita OOM killer no Redis

Plano de Rollback

Reativar Redis Cloud: alterar variável REDIS_URL no .env do n8n de redis://127.0.0.1:6379 para a URL original do ElastiCache. Docker local pode ficar rodando em paralelo sem conflito — as portas não colidem (Redis cloud fica na porta remota padrão, Redis local em 127.0.0.1:6379).
