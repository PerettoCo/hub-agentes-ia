Timeout + Kommo 404 · Resolvido

O SDR IA da ADPLAN parava de responder no WhatsApp. A causa era dupla: timeout de 5 minutos matava execuções longas, e um erro 404 ao atualizar lead no Kommo derrubava todo o fluxo.

Resumo Simples

Esta seção explica o problema de forma clara, sem jargão técnico. Ideal para entender o que aconteceu e qual foi o impacto.
!

O problema: O robô de vendas (SDR IA) que atende leads no WhatsApp para a ADPLAN parava de responder no meio da conversa. Clientes ficavam sem resposta e o atendimento morria.

1Timeout muito curto
O robô tinha apenas 5 minutos para processar cada atendimento. Quando o lead demorava para responder ou a conversa era longa, o robô simplesmente desistia — como um cronômetro estourando. Aumentamos para 20 minutos, tempo mais que suficiente para qualquer atendimento.
2Erro ao salvar no Kommo
No final do atendimento, o robô tenta atualizar o lead no sistema Kommo (CRM). Mas ele estava tentando atualizar sem o ID do lead — como se você tentasse editar um documento sem saber o número dele. O sistema devolvia erro e derrubava toda a conversa. Agora ele ignora esse erro e continua normalmente.
✓

Resultado: O SDR IA agora completa todos os atendimentos sem morrer no meio. Mesmo que a atualização no Kommo falhe, a conversa no WhatsApp continua normalmente. Clientes não percebem diferença.

Investigação

Análise da Causa Raiz

Diagnóstico completo baseado na análise de execuções com erro e engenharia reversa do workflow.

Fluxo do erro

1 WhatsApp Trigger
›
2 AI Agent (VendedorRemo1)
›
✗ HTTP Request1 — PATCH Kommo
›
✗ Fluxo morto (sem resposta)

Causa #1 — Timeout insuficiente: O workflow estava configurado com executionTimeout: 300 (5 minutos). Cada execução do SDR IA envolve múltiplas chamadas de API (OpenAI, Supabase, Redis, Kommo) e aguarda respostas do usuário via WhatsApp. Em conversas mais longas ou com lead demorando para responder, os 5 minutos estouravam antes do fim do fluxo, matando a execução.

Causa #2 — continueOnFail ausente: O nó HTTP Request1 faz um PATCH para https://rogerioadplanengenhariacombr.kommo.com/api/v4/leads/{{$json.id}}. O campo $json.id deveria vir do nó anterior (VendedorRemo1 — AI Agent), mas o AI Agent não retorna o ID do lead no formato esperado. O resultado é a URL /api/v4/leads/ (sem ID), que retorna 404. Sem continueOnFail: true, esse 404 propaga como erro fatal para todo o workflow.

i

Evidência: Execução #233926 (19 Jun 00:54 UTC) — erro Cannot PATCH https://rogerioadplanengenhariacombr.kommo.com/leads/! — confirmando que $json.id estava vazio.

Correção

O que Mudou

Duas alterações cirúrgicas no workflow via API REST direta. Nenhuma mudança estrutural — apenas parâmetros.

Complexidade da correção via API

O desafio técnico real não foi identificar o bug (claro desde a primeira execução com erro), mas sim conseguir aplicar a correção via API. O PUT do n8n faz validação estrita de todos os nós, mesmo os que nunca executam. Foram necessárias 4 tentativas:

Tentativa 1: PUT com apenas name + nodes + connections + settings — rejeitado (settings com props extras)

Tentativa 2: PUT com settings mínimos — rejeitado (8 nós com config issues)

Tentativa 3: PUT com settings mínimos + correção dos 8 nós (usando __rl params) — rejeitado (Google Drive fileId + Supabase Vector Store tableName)

Tentativa 4: PUT com settings mínimos + correção 8 nós (plain string params + expressão) — ✅ aceito
