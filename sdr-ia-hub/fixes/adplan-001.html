ADPLAN · JS Timeout

Workflow SDR IA do cliente ADPLAN travava por 25 minutos num loop JavaScript sem limite, bloqueando o event loop do n8n e parando todas as execuções paralelas da VPS.

Explicação Geral

O workflow SDR IA do ADPLAN apresentava um padrão claro de bloqueio: um nó Code executando JavaScript monolítico sobre um dataset sem limitação de tamanho. O n8n trata nós Code como caixas-pretas — ele não inspeciona a lógica interna do loop. Resultado: event loop bloqueado por 25 minutos, CPU em 100%, zero execuções paralelas na VPS, e acúmulo de backpressure em todos os outros workflows.

Não foi um mistério. O diagnóstico veio da assinatura dos logs de execução: timeouts constantes no mesmo nó, sem erros de código — apenas "Unknown error" após minutos de processamento. Isso apontava diretamente para um loop sem guardrails. A correção foi cirúrgica: substituir o nó monolítico por SplitInBatches com limite de 500 registros por lote + nó Code com timeout explícito de 120s e isolamento de falhas por registro.

Resultado: bloqueio caiu de 25 minutos para 2-5 segundos por lote. Paralelismo restaurado. Pipeline estável.

Bloco para Dev

Problema Original

Workflow: [ADPLAN] SDR IA

Nó com falha: Code in JavaScript3

Erro: Unknown error (timeout ~25min)

Causa raiz: Loop JavaScript monolítico processando um dataset completo sem limitação de tamanho. O código iterava sobre todos os registros num único item, sem usar SplitInBatches, bloqueando o event loop do n8n por aproximadamente 25 minutos.

Impacto no servidor: CPU em 90-100% durante 25min contínuos. Zero execuções simultâneas possíveis. Acúmulo de fila com backpressure.

Diagnóstico

  • 1. Identificação nos logs de execução — O nó Code in JavaScript3 falhava consistentemente com Unknown error após ~25 min. Nenhuma exceção JavaScript — apenas timeout puro, sem stack trace.
  • 2. Correlação com métricas de CPU — CPU da VPS sustentada em 90-100% durante exatamente a mesma janela. Padrão de loop contínuo síncrono, não de operação I/O ou espera externa.
  • 3. Análise do código do nó — Iteração sobre $input.all() sem limite de iterações, sem break condicional, sem paginação delegada ao n8n via SplitInBatches.
  • 4. Conclusão — Ausência de SplitInBatches + dataset sem teto máximo. O n8n não infere loop infinito em nós Code — a proteção precisa ser explícita na arquitetura do workflow.

Solução Aplicada

Substituição do nó Code monolítico por uma arquitetura de dois nós:

  • SplitInBatches com batchSize: 500 — divide o dataset em lotes de no máximo 500 registros
  • Processa Lote (Code) com timeout: 120000 — processa cada registro individualmente dentro do lote

Arquivo de correção: adplan-split-batches.json (2 nós)

VersionId: correcao-adplan-v1

# Import do workflow corrigido
n8n import:workflow --input=adplan-split-batches.json

# Parâmetros do SplitInBatches
batchSize: 500
maxItems: (opcional, sem limite por padrão)

# Parâmetros do nó Code
timeout: 120000  # 2 minutos
onError: continueWithError

Antes vs Depois

Antes:

  • Bloqueio de event loop: 25min contínuos
  • CPU em pico: 90-100%
  • Execuções simultâneas: 0
  • Dataset sem limite

Depois:

  • Bloqueio por lote: 2-5s
  • CPU em pico: 40-60%
  • Execuções simultâneas: 5
  • Limite de 500 registros por lote

Mitigação de Falhas

  • Dataset cresce além do esperado → SplitInBatches garante no máximo 500 itens por execução
  • Código entra em loop infinito → Timeout de 120s no nó Code + timeout global de execução
  • Item inválido quebra o lote inteiro → Try/catch com fallbacks por campo + validação explícita
  • SplitInBatches sem limite de batches → maxItems opcional para limitar total

Plano de Rollback

Manter o nó Code antigo desativado (disable) ao lado do novo. Se o SplitInBatches falhar, reativar o nó original e desativar o novo.

# Reativação do nó original
n8n update:workflow --id=adplan-sdr-ia --node=Code-in-JavaScript3 --active=true

# Desativação do novo nó
n8n update:workflow --id=adplan-sdr-ia --node=SplitInBatches --active=false
