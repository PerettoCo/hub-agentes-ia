PRO ANALISES · toDateTime

Explicação Simplificada

O que aconteceu?

O robô do PRO ANALISES precisa saber quando foi a última atividade de cada lead para decidir se deve ou não entrar em contato. Esse campo (ultimaatividade) é como uma "data da última conversa".

O problema é que, para alguns leads que nunca tiveram atividade, esse campo simplesmente não existe — vem vazio. O robô tentava ler essa data, não achava nada (undefined), e travava inteiro.

Imagine alguém perguntar "que horas foi a última reunião?" para um lead que nunca teve reunião. A resposta é "não sei" — mas o robô não aceita "não sei" como resposta, ele simplesmente para de funcionar.

O que mudou?

Colocamos um "tradutor" antes do robô tentar usar essa data. Esse tradutor verifica: o campo existe? Se sim, usa ele. Se não, inventa uma data padrão: "dois dias atrás".

É como aquele amigo que quando não sabe a resposta, chuta "ontem" só pra conversa não morrer. Só que aqui o chute é inteligente — 48h atrás é um fallback seguro que não atrapalha o resto do processo.

Resultado esperado

Zero erros de toDateTime. Leads sem atividade recente entram no fluxo normal com a data de fallback. A prospecção não trava mais por falta de dado.

Detalhamento Técnico

Problema Original

Workflow: [PRO ANALISES] SDR IA

Nó com falha: IF (Filter by ultimaatividade)

Erro: toDateTime can't be used on undefined value

Causa raiz: O trigger do workflow (webhook ou schedule) retornava objetos lead onde o campo ultimaatividade estava ausente (undefined) para leads sem interação prévia. O nó IF usava $toDateTime(ultimaatividade) sem validação, e a engine do n8n rejeita undefined no parser de datas.

Impacto: Toda execução que incluía pelo menos um lead sem ultimaatividade falhava com erro não recuperável. Dependendo do volume, parava o workflow inteiro.

Solução Aplicada

Inserir nó Code de sanitização entre o trigger e o IF, que normaliza o campo ultimaatividade antes do filtro:

const items = $input.all();
const result = items.map(item => {
  const lead = item.json;
  const fallback = new Date(Date.now() - 48 * 60 * 60 * 1000).toISOString();
  return {
    json: {
      ...lead,
      ultimaatividade: lead.ultimaatividade || fallback,
      _sanitized: !lead.ultimaatividade
    }
  };
});
return result;

Antes vs Depois

Antes:

  • $toDateTime(undefined) → crash
  • Lead sem atividade quebrava tudo
  • Zero fallback no fluxo
  • Erro não recuperável

Depois:

  • Sanitização com fallback de 48h
  • Campo _sanitized para auditoria
  • Zero erros de tipo
  • Logs rastreáveis

Mitigação de Falhas

  • Campo ultimaatividade chega como null → fallback cobre também ||
  • Campo chega como string inválida → não falha, $toDateTime tenta parsear
  • Trigger muda formato → sanitização é adaptável, fácil de ajustar regex
  • Fallback excessivo em muitos leads → campo _sanitized permite monitorar proporção de fallbacks

Plano de Rollback

Desabilitar o nó Code de sanitização e reconectar o trigger diretamente no IF. O erro original vai voltar, mas o fluxo anterior fica preservado.
