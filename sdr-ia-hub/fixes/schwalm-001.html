SCHWALM · Null Telefone

Contatos vindos do WhatsApp sem telefone cadastrado quebravam a query PostgreSQL com null value in column "telefoneCliente" violates not-null constraint. Um nó Code de validação + IF agora filtra esses itens antes do insert.

Explicação Geral

Contexto do Problema

Durante a importação de contatos via WhatsApp no fluxo do Schwalm, identificamos que a API de mensageria (Quepasa) retornava objetos de contato sem o campo telefoneCliente preenchido — contatos podiam vir identificados exclusivamente pelo waId (WhatsApp ID), sem um número de telefone associado. Ao tentar inserir esses registros no PostgreSQL, a constraint NOT NULL da coluna telefoneCliente rejeitava o batch inteiro, resultando em falha mesmo quando apenas um contato do lote era inválido. O nó PostgreSQL do n8n executa inserts em lote — um único item nulo invalida a transação completa.

Não se tratava de um erro de configuração ou de infraestrutura. Era um problema de validação de entrada ausente: o fluxo não possuía qualquer saneamento dos dados antes do INSERT. Sabíamos exatamente o que procurar porque o erro no log era explícito — null value in column "telefoneCliente" violates not-null constraint — e a intermitência (lotes passavam quando todos os contatos tinham telefone) confirmava que a causa era os dados, não o banco.

Correção Realizada

Implementamos um nó Code de validação posicionado antes do INSERT, responsável por inspecionar cada item e separar contatos válidos (com telefone) de inválidos (sem telefone). Um nó IF roteia os válidos para o PostgreSQL e os inválidos para um nó de log. Nenhuma alteração na estrutura da tabela ou no esquema do banco foi necessária — a solução respeita as constraints existentes e adiciona uma camada de saneamento na aplicação, que é onde esse tipo de validação deve estar.

Desde a correção, zero erros de not-null constraint foram registrados. Contatos sem telefone são isolados e logados com _motivo_rejeicao, permitindo tratamento posterior (manual ou por outro fluxo) sem interromper a importação dos contatos válidos.

Bloco para Desenvolvedores

Problema Original

Workflow: [SCHWALM] SDR IA — Importação de Contatos

Nó com falha: PostgreSQL (Insert)

Erro: null value in column "telefoneCliente" violates not-null constraint

Causa raiz: A integração com WhatsApp (via Quepasa ou API oficial) retornava objetos de contato onde telefoneCliente era null — contatos identificados apenas por waId sem número de telefone associado. O nó PostgreSQL tentava inserir o registro sem o campo obrigatório, e a constraint NOT NULL da tabela rejeitava a operação inteira.

Impacto: Se qualquer contato de um lote tivesse telefone nulo, o INSERT inteiro falhava (não apenas o registro problemático), porque o nó PostgreSQL do n8n executava um batch insert.

Diagnóstico

Cadeia de identificação do problema:

  1. Log de execução (n8n): O nó PostgreSQL apresentava falha intermitente com a mensagem null value in column "telefoneCliente" violates not-null constraint.
  2. Análise de padrão: A falha não era consistente — lotes com todos os contatos válidos passavam sem erro. Isso eliminava hipóteses de configuração do banco ou da conexão.
  3. Inspeção do payload upstream: Abrimos a saída do nó anterior (Webhook/API WhatsApp) e confirmamos: contatos sem telefoneCliente retornavam null no campo, mas com waId preenchido. O WhatsApp ID existe independentemente do número de telefone.
  4. Entendimento do batch insert: O nó PostgreSQL do n8n executa INSERT em lote. Um único item nulo invalida o lote inteiro por causa da constraint NOT NULL — a transação é atômica.
  5. Reprodução controlada: Enviamos propositalmente um contato sem telefone e reproduzimos o erro com a mesma mensagem, confirmando a hipótese.

Conclusão do diagnóstico: Validação de entrada ausente. O fluxo não sanitizava os dados antes do INSERT, e o banco rejeitava o batch por constraint violada.

Solução Aplicada

Inserir nó Code de validação antes do PostgreSQL, seguido de um IF router:

const items = $input.all();
const validos = [];
const invalidos = [];

items.forEach(item => {
  const c = item.json;
  if (c.telefoneCliente && c.telefoneCliente.toString().trim()) {
    validos.push({ json: c });
  } else {
    invalidos.push({
      json: {
        ...c,
        _motivo_rejeicao: 'telefoneCliente ausente ou vazio',
        _rejeitado_em: new Date().toISOString()
      }
    });
  }
});

// Saída 0 (válidos) → segue para PostgreSQL
// Saída 1 (inválidos) → log para debug
return [validos, invalidos];

Fluxo corrigido:

  • Entrada: Webhook do WhatsApp com lista de contatos
  • Nó Code (validação): Separa válidos de inválidos
  • IF router: Válidos → PostgreSQL Insert | Inválidos → Log + nó NoOp

Arquivo de correção: schwalm-validacao-telefone.json (2 nós: Code + IF)

VersionId: correcao-schwalm-v1

Antes vs Depois

Antes:

  • Batch insert falhava com 1 item inválido
  • Contato sem telefone quebrava o lote
  • Zero visibilidade sobre rejeições
  • Log: null value violates not-null constraint

Depois:

  • Válidos inseridos, inválidos isolados
  • Zero falhas de constraint
  • Campo _motivo_rejeicao rastreável
  • Log claro sobre cada rejeição

Mitigação de Falhas

  • Campo telefoneCliente vem como null → validação captura com && .toString().trim()
  • Campo vem como string vazia "" → .trim() captura
  • Campo vem como número 0 → falso positivo evitado por .toString()
  • Todos inválidos → fluxo não quebra, apenas loga

Plano de Rollback

Remover os nós Code e IF, reconectando a saída do trigger direto ao PostgreSQL. O erro original voltará se houver contatos sem telefone.
