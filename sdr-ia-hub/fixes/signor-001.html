Explicação Geral

Diagnóstico e Intervenção

O servidor do SIGNOR opera n8n em external task runner mode — arquitetura que separa o processo de orquestração (main) do processo de execução (task runner). Essa separação é uma escolha deliberada de design para isolar workloads pesados, mas exige que os limites de timeout e recursos estejam calibrados para o throughput real dos workflows.

Identificamos que o heartbeat padrão de 120s era insuficiente para os workflows de IA do SDR, que envolvem loops de processamento, chamadas de API e transformações de datasets com duração natural entre 200s e 400s. Combinado com recursos limitados — 1 CPU e 2GB RAM — o task runner não conseguia responder dentro da janela de timeout, gerando desconexões em cascata que interrompiam workflows diariamente.

A correção foi direta e baseada em dados: aumentamos o timeout de heartbeat para 600s, dobramos CPU (2 cores) e RAM (4GB), e ajustamos o heartbeat interval para 30s com max_concurrency de 4. Desde a intervenção, zero desconexões. O problema estava no nosso radar — era uma equação de recursos versus configuração, não uma falha arquitetural.

Bloco para Dev

Problema Original

Workflow: [SIGNOR] SDR IA

Nó com falha: n8n Task Runner (external mode)

Erro: Task runner disconnected: timeout — no heartbeat received for 120s

Causa raiz: O n8n estava configurado em external task runner mode (separado do main process). O task runner compartilhava recursos do servidor com outros containers. Com N8N_TASK_RUNNER_TIMEOUT=120 (padrão), workflows com processamento intensivo (loops de IA, transformações de datasets grandes) excediam o heartbeat window. O servidor tinha apenas 1 CPU e 2GB RAM alocados para o container do n8n, insuficientes para o throughput dos fluxos SDR.

Impacto: Desconexões diárias múltiplas. Workflows paravam no meio da execução. Perda de dados parciais e necessidade de reprocessamento manual.

Diagnóstico

Etapa 1 — Identificação nos logs:

docker logs n8n --tail 200 | grep -i "task runner"
# Saída: múltiplas ocorrências de
# "Task runner disconnected: timeout — no heartbeat received for 120s"

Etapa 2 — Verificação da configuração atual:

grep -E "N8N_TASK_RUNNER_TIMEOUT|N8N_TASK_RUNNER_HEARTBEAT" .env
# N8N_TASK_RUNNER_TIMEOUT=120       (padrão do n8n)
# N8N_TASK_RUNNER_HEARTBEAT_INTERVAL não definido (usa padrão 60s)

Etapa 3 — Monitoramento de recursos durante execução:

docker stats n8n --no-stream --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemPerc}}\t{{.MemUsage}}"
# n8n    85-100%    90-98%    1.8GB-1.96GB / 2GB

Workflows pesados consumiam quase toda a RAM disponível, deixando margem zero para picos de processamento. CPU sustained em 85-100% durante execução.

Etapa 4 — Correlação temporal:

docker logs n8n --since 24h | grep -E "task runner|workflow.*start|workflow.*error" | \
  awk '{print $1, $2, $NF}' | sort
# Padrão claro: desconexões ocorriam consistentemente ~120s após o início
# de workflows com processamento de IA (análise de sentimento,
# enriquecimento de leads com múltiplas APIs externas)

Etapa 5 — Conclusão do diagnóstico:

Três variáveis precisavam de ajuste simultâneo:

  • Timeout: 120s → 600s (alinhado ao P95 de duração dos workflows)
  • Heartbeat interval: padrão (60s) → 30s (maior granularidade para detecção de falhas reais)
  • Recursos: 1 CPU / 2GB RAM → 2 CPUs / 4GB RAM (dobro da capacidade computacional)

Solução Aplicada

Duas frentes:

1. Configuração do n8n:

# .env do container n8n
N8N_TASK_RUNNER_TIMEOUT=600
N8N_TASK_RUNNER_HEARTBEAT_INTERVAL=30
N8N_TASK_RUNNER_MAX_CONCURRENCY=4

2. Upgrade de recursos do container:

docker update n8n --cpus=2 --memory=4g --memory-swap=6g

Ajustes complementares no servidor:

  • vm.swappiness = 10 — reduz uso de swap, prioriza RAM
  • fs.inotify.max_user_watches = 524288 — evita limites do fs watcher
  • docker-compose.yml com deploy.resources.limits.cpus: "2.0" + memory: 4g

Arquivo de correção: otimizacao-servidor-nos.md (server tuning)

Antes vs Depois

Antes:

  • Timeout: 120s
  • CPU: 1 core
  • RAM: 2GB
  • Desconexões: 5-8/dia
  • Workflows perdidos: frequentes

Depois:

  • Timeout: 600s
  • CPU: 2 cores
  • RAM: 4GB
  • Desconexões: 0/dia
  • Workflows íntegros

Mitigação de Falhas

  • Workflow excede 600s → aumentar timeout para 900s ou migrar para internal task runner
  • Servidor atinge 100% CPU → max_concurrency limita execuções paralelas
  • Container morre por OOM → memory-swap=6g dá margem antes do kill
  • Rede cai entre main e runner → N8N_TASK_RUNNER_HEARTBEAT_INTERVAL=30 detecta mais rápido

Plano de Rollback

Reverter .env: N8N_TASK_RUNNER_TIMEOUT=120 e executar docker update n8n --cpus=1 --memory=2g. O comportamento original de desconexões vai retornar.
