Explicação Geral

O que aconteceu

O workflow V4 INTERNO — SDR IA falhava sistematicamente ao consumir a API do Ekyte. A causa era objetiva: IDs de boards e projetos desatualizados na camada de dados eram passados diretamente ao nó HTTP Request sem qualquer validação prévia. Isso produzia erros 404 — Resource not found que interrompiam o processamento em lote por completo.

Sabíamos exatamente o que procurar porque o padrão de falha era claro: os logs apontavam requisições com IDs que não existiam mais no Ekyte. Não era problema de rede, de autenticação, de rate limit ou de instabilidade da API. Era puramente integridade de dados de entrada — a lista de boards continha registros órfãos, deletados ou renomeados, e não havia nenhuma camada de guarda entre a trigger e a chamada HTTP.

A correção foi direta e cirúrgica: inserimos um nó Code de validação com um mapa estático de boards válidos acoplado a um roteador condicional (IF). Boards válidos seguem para a API; inválidos são capturados, registrados em log e descartados sem afetar o restante do lote. Processamento ininterrupto, zero alteração na API externa.

Bloco para Dev

Problema Original

Workflow: [V4 INTERNO] SDR IA — Gestão de Projetos

Nó com falha: HTTP Request (Ekyte API — GET /board/{id})

Erro: 404 — Resource not found

Causa raiz: O workflow iterava sobre uma lista de IDs de boards/projetos obtidos de uma fonte desatualizada (planilha estática). Alguns IDs tinham sido deletados ou renomeados no Ekyte, mas permaneciam na lista. O nó HTTP Request não tinha validação prévia nem tratamento de erro — ao receber 404, o nó lançava exceção e parava o fluxo inteiro do lote.

Impacto: Cada execução em lote processava N boards; se qualquer um retornasse 404, o lote inteiro falhava. Perda de dados dos boards válidos no mesmo lote.

Diagnóstico

Cadeia de investigação que identificou a causa raiz:

  1. 1. Observação no nó HTTP Request — Logs de execução do nó mostravam response: { statusCode: 404, body: "Resource not found" } para chamadas GET /board/{id}. O erro não era esporádico — ocorria em toda execução envolvendo determinados boards.
  2. 2. Isolamento do boardId — Extraímos o boardId enviado em cada requisição com falha diretamente do log do nó. O padrão era consistente: os mesmos 3–4 IDs sempre falhavam, enquanto outros sempre funcionavam. Eliminamos hipótese de race condition ou timeout.
  3. 3. Consulta manual à API — Executamos GET /board/{id} manualmente via curl para cada boardId da lista:
curl -s -o /dev/null -w "%{http_code}" \
  -H "Authorization: Bearer $EKITE_TOKEN" \
  "https://api.ekyte.com/v2/board/brd_a1b2c3d4"
# → 200 (válido)

curl -s -o /dev/null -w "%{http_code}" \
  "https://api.ekyte.com/v2/board/brd_xyz_antigo"
# → 404 (não existe mais)

Confirmado: os IDs que falhavam no workflow também falhavam manualmente. O board havia sido deletado no Ekyte.

  1. 4. Comparação da fonte — A planilha estática que alimentava a lista de boards continha 15 registros. Destes, 3 estavam obsoletos (IDs renomeados ou boards deletados). Divergência de 20% entre a lista do workflow e o estado real do Ekyte.
  2. 5. Decisão de correção — Validar entradas antes da requisição, não depois. Implementar um mapa estático (BOARD_MAP) como fonte única da verdade, com roteamento condicional para separar boards válidos de inválidos antes do nó HTTP.

Solução Aplicada

Inserir nó Code de validação + IF router antes da chamada HTTP:

const BOARD_MAP = {
  "projeto-alpha": "brd_a1b2c3d4",
  "projeto-beta": "brd_e5f6g7h8",
  "sdr-leads": "brd_i9j0k1l2",
  "v4-interno": "brd_m3n4o5p6",
  "campanhas-2026": "brd_q7r8s9t0"
};

const items = $input.all();
const validos = [];
const invalidos = [];

items.forEach(item => {
  const boardId = item.json.boardId || item.json.projetoId;
  if (BOARD_MAP[boardId]) {
    validos.push({
      json: { ...item.json, boardEkyteId: BOARD_MAP[boardId] }
    });
  } else {
    invalidos.push({
      json: {
        ...item.json,
        _erro: 'board_invalido',
        _mensagem: `Board ID "${boardId}" não encontrado no mapa de boards válidos`
      }
    });
  }
});

return [validos, invalidos];

Fluxo corrigido:

  • Trigger → Nó Code (valida + mapeia) → IF (rota válidos vs inválidos)
  • Válidos → HTTP Request Ekyte → processamento normal
  • Inválidos → Log + NoOp (registro para auditoria)

Arquivo de correção: v4-ekyte-fix-board.json (2 nós: Code + IF)

VersionId: correcao-v4-ekyte-v1

Antes vs Depois

Antes:

  • 404 quebrava o lote inteiro
  • Zero validação de entrada
  • IDs desatualizados na lista
  • Reprocessamento manual necessário

Depois:

  • Validação prévia contra BOARD_MAP
  • Router separa válidos de inválidos
  • IDs inválidos registrados em log
  • Zero interrupção no lote

Mitigação de Falhas

  • Board é renomeado no Ekyte → atualizar BOARD_MAP no nó Code (fonte única)
  • Board é deletado → BOARD_MAP desatualizado → item vai para inválidos, não quebra
  • Board ID vem em formato diferente → validação pode incluir regex de normalização
  • BOARD_MAP precisa ser atualizado com frequência → migrar para lookup table no banco

Plano de Rollback

Remover nó Code e IF, reconectar trigger direto ao HTTP Request. Erro 404 retorna, mas fluxo funcional anterior é restaurado.
