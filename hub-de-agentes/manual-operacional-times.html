O Conceito: Orquestradores e Especialistas

O Agent Hub V4 não é uma lista plana de 36 agentes. É um sistema operacional de times onde agentes orquestradores coordenam agentes especialistas — exatamente como um líder de squad coordena seu time.

⚙️ Antes de tudo, conheça as Funções Nativas: O Builders Hub vem com comandos base que são o "sistema operacional" do repositório. /onboarding configura tudo na primeira vez. /contexto atualiza o agente sobre o cliente (use sempre que começar/terminar trabalho). /sabatina testa planos. /sync-hub puxa skills novas. Veja a seção 02 para o fluxo completo.

A Hierarquia

Como Funciona na Prática

O usuário nunca fala diretamente com um especialista. Ele fala com o orquestrador, que monta o time certo para cada tarefa:

Usuário: "@account-orchestrator, acabei a call com o cliente X. Processa o transcript."

Orquestrador faz:
1. Chama @account-checkin-review para processar o transcript
2. Extrai combinados → chama Ekyte MCP para criar tasks
3. Se detectou risco → chama @flag-churn para diagnóstico
4. Pede @revisor para validar tudo
5. Entrega resultado consolidado ao usuário

Por Que Isso É Poderoso

Sem Times (plano)Com Times (orquestrado)
Usuário precisa saber qual agente chamarUsuário fala com um orquestrador — ele monta o time
Cada agente trabalha isoladoAgentes colaboram em cadeia
Sem garantia de qualidade@revisor valida antes de entregar
Contexto se perde entre agentesOrquestrador mantém o contexto do início ao fim
Usuário gerencia o fluxo manualmenteOrquestrador gerencia o fluxo automaticamente
Cada um usa o modelo que querOrquestrador escolhe o modelo certo para cada tarefa
🔑 A regra de ouro: O orquestrador pensa e decide. O especialista executa. O revisor valida. O Ekyte registra. O humano aprova.

Funções Nativas do Builders Hub — O Sistema Operacional

Antes de qualquer skill especializada, existem as funções nativas do Builders Hub. Elas vêm da matriz e são o "sistema operacional" do repositório — cuidam de setup, contexto, estrutura e governança. São comandos /slash que você roda no chat do agente.

⚙️ Regra de ouro: Sempre que terminar uma seção de trabalho com um cliente, rode /contexto. O agente precisa do conhecimento atualizado para funcionar bem. Se for a primeira vez no repo, rode /onboarding antes de tudo.

Fluxo Operacional Recomendado

🆕 Primeira vez no repositório → /onboarding
Valida git/gh 100%, instala dependências (Node, Python, Claude Code), ensina a estrutura do Builders Hub. Roda sempre que algo do setup quebrar.
📥 Antes de começar o dia → /sync-hub
Puxa skills mais recentes do repositório público, mostra diff do que mudou, regenera REGISTRY.md. Recomendado semanalmente.
📁 Trabalhando em um cliente → /contexto
Lê toda a KB do cliente, gera CLAUDE.md e AGENTS.md, atualiza Mission Control com OKRs, apostas, combinados e personas. Rode antes de começar e depois de terminar.
🧠 Testando um plano → /sabatina
Stress-test implacável de planos e decisões. O agente percorre cada ramo da árvore de decisão até atingir entendimento compartilhado.
🚀 Criando cliente/squad → /novo-squad, /novo-cliente, /novo-projeto
Criam a estrutura de pastas padronizada. Novo squad pede membros. Novo cliente pede squad e links úteis. Novo projeto cria base genérica.
💡 Descobrindo onde a IA ajuda → /geral-brainstormar
Entrevista sobre seu dia a dia para mapear onde IA agrega mais valor. Sugere skills e atualiza seu perfil.
🔧 Criando skill própria → /criador-de-skills
Transforma um workflow que funcionou em skill reutilizável. Prefixo obrigatório. Duplo-write automático.
📤 Compartilhando com o time → /compartilhar-skill
Empacota a skill, valida frontmatter, resolve conflitos com origin/main, cria branch e abre PR via gh CLI.
💾 Salvando a sessão → /session-save
Exporta a sessão atual para log/ como JSON. Use ao final de cada interação significativa para continuar depois.

Opensquad — O Backbone de Orquestração

O Opensquad é o gestor dos squads de agentes. Ele cria, configura, executa pipelines e mantém memória entre sessões. É o maestro que coordena os 8 times e seus especialistas.

CapacidadeComo Funciona
Onboarding de empresaColeta dados, pesquisa website, gera company.md com perfil completo
Criação de SquadsCria squad com pipeline (squad.yaml), agentes (squad-party.csv) e memória persistente
Execução de PipelineLê squad.yaml, carrega agentes do CSV, executa com checkpoints
MemóriaRegistra aprendizados em memories.md após cada execução
15+ Comandos/opensquad, create, run, list, edit, delete, skills, install, settings

Como Opensquad e Skills se Conectam

⚡ A diferença: Opensquad é o gestor — cria squads, executa pipelines, mantém memória. As skills são o conhecimento — o "como fazer" de cada tarefa. Opensquad sem skills é maestro sem orquestra. Skills sem Opensquad são instrumentos sem músico.

Cadeia de Valor: Como a Eficiência Acontece

O ganho real não está nos agentes individuais — está em como eles se combinam. Cada orquestrador monta o time certo para cada tipo de demanda, eliminando retrabalho e garantindo consistência.

Matriz de Invocação

Quem chama quem na operação V4:

📋 Account quer processar check-in

@account-orch → @checkin-review → Ekyte MCP → @revisor

🎯 CMO quer estratégia de marketing

@cmoorch → @analista-dados → @estrategia-marketing → @revisor

✍️ Content Studio quer produzir conteúdo

@content-studio → @pesquisador → @copy-content → @criacao-design → @seo-visibilidade → @revisor

📈 Growth Team quer plano integrado

@growth-team → @analista-dados → @cro-otimizacao → @midia-paga → @seo-visibilidade

🛡️ CSM quer diagnóstico de cliente

@csm-orquestrador → @flag-churn → @flag-okr → @flag-operacao → @account-orch

🚀 Launch Pad quer lançar produto

@launch-pad → @estrategia-marketing → @copy-content → @midia-paga → @seo-visibilidade

O Fluxo de Valor em 3 Camadas

🎯 Entrada: Demanda chega ao orquestrador
Usuário fala com o orquestrador certo. Ele interpreta a demanda, decide o time necessário e o modelo ideal para cada parte.
Ex: "@account-orchestrator" Ex: "@cmoorch"
🧠 Processamento: Orquestrador monta o time
Orquestrador convoca especialistas em sequência ou paralelo, passa contexto, define prazos. Cada especialista usa o MCP certo + modelo ideal.
@checkin-review → Ekyte MCP → @revisor
📤 Saída: Resultado consolidado no Ekyte
Orquestrador consolida tudo, registra no Ekyte (tasks, comentários, tickets), salva artefatos no Drive, e entrega ao usuário com links.
create_task create_task_comment create_file (Drive)

Regras de Engajamento

  1. Nunca pule o orquestrador. Usuário não fala diretamente com especialista — o orquestrador monta o time certo.
  2. Sempre passe contexto. Orquestrador nunca invoca especialista sem briefing claro do que precisa.
  3. Sempre valide. @revisor antes de entregar para o usuário. Sem exceção.
  4. Sempre registre. Toda ação no Ekyte. Comentário explicando origem. Link para artefato no Drive.
  5. Um orquestrador por demanda. Se @cmoorch acionou @content-studio, o Content Studio orquestra os especialistas — o CMO não micro-gerencia.
  6. Flag é ticket. Toda flag detectada vira ticket no Ekyte com prioridade e responsável.
  7. Nunca duplique esforço. Antes de invocar, verifique se já existe task/comentário sobre o assunto.
