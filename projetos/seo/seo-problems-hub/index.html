01 SSL e TLS — certificado que bloqueia a indexação

Crítico Infraestrutura

Visão geral

O Googlebot recusa conexões HTTPS com certificados expirados, autoassinados, com hostname mismatch ou que usam protocolos TLS obsoletos. Quando o crawler encontra qualquer um desses problemas, ele aborta o crawl: não estabelece conexão segura, não baixa o HTML, não indexa a página. O site simplesmente desaparece do índice.

Diferente de um usuário que pode clicar em "Prosseguir mesmo assim", o Googlebot não contorna avisos de segurança. Se o handshake TLS falha, a URL é tratada como inacessível.

Causa raiz

O protocolo TLS exige que o servidor apresente um certificado válido emitido por uma Autoridade Certificadora (CA) confiável. O Googlebot usa o mesmo trust store do Chrome: se o Chrome rejeita, o Googlebot rejeita.

CenárioComportamento do Googlebot
Certificado expiradoRecusa a conexão — ERR_CERT_DATE_INVALID
Autoassinado ou CA desconhecidaRecusa — ERR_CERT_AUTHORITY_INVALID
Hostname mismatchCert emite para www.site.com, site responde em site.com
TLS 1.0 / 1.1Googlebot não aceita protocolos obsoletos desde 2020
Cadeia incompletaCertificado intermediário ausente — handshake falha
HTTP para HTTPS quebradoRedirect loop ou URL HTTPS que não resolve

Impacto no SEO

  • Desindexação total — páginas HTTPS com TLS inválido somem do índice
  • Crawl budget desperdiçado — Googlebot gasta recursos tentando e falhando
  • Aviso "Não seguro" no navegador — impacta taxa de rejeição e confiança
  • Redirecionamento HTTP para HTTPS quebrado — usuários e crawlers caem em página de erro
  • Penalidade indireta — Google trata sites sem HTTPS como não confiáveis

Diagnóstico

1. SSL Labs Test

Acesse https://www.ssllabs.com/ssltest/ e insira a URL. O relatório mostra validade, cadeia de confiança, versão TLS, ciphers, HSTS e nota geral (A+ a F).

2. Google Search Console

URL Inspection: veja se o Googlebot reporta erro de "Crawled but blocked" ou "Page not indexed". Expanda "Crawling" para ver detalhes do HTTP response.

3. curl com user-agent do Googlebot

curl -I -A "Mozilla/5.0 (compatible; Googlebot/2.1; \
  +http://www.google.com/bot.html)" \
  -H "Accept-Language: en-US,en;q=0.9" \
  https://seu-site.com

4. openssl (teste direto do certificado)

echo | openssl s_client -connect seu-site.com:443 \
  -servername seu-site.com 2>/dev/null | \
  openssl x509 -noout -dates

Solução

ProblemaSolução
ExpiradoRenovar com a CA e instalar antes do vencimento. Configurar alerta 30 dias antes
AutoassinadoSubstituir por certificado de CA confiável (Let's Encrypt, Sectigo, Cloudflare)
Hostname mismatchReemitir certificado SAN cobrindo todos os domínios
TLS obsoletoDesabilitar TLS 1.0 e 1.1. Manter apenas TLS 1.2 e 1.3
Cadeia incompletaInstalar o certificado intermediário no servidor
Redirect loopCorrigir configuração do servidor web para redirect 301 direto

POP — Procedimento Operacional Padrão

  1. Diagnosticar com SSL Labs, capture a nota e os problemas
  2. Validar no GSC — URL Inspection, veja se o Googlebot reporta erro de TLS
  3. Identificar o tipo de falha: expirado, mismatch, autoassinado, cadeia ou TLS
  4. Se expirado: gerar novo CSR, comprar ou renovar com a CA, instalar cert + chain
  5. Se autoassinado: usar Let's Encrypt via Certbot: certbot --nginx -d site.com -d www.site.com
  6. Se mismatch: reemitir certificado SAN incluindo todos os domínios e subdomínios
  7. Se cadeia incompleta: baixar o bundle CA + intermediário e concatenar ao certificado
  8. Se TLS obsoleto: no Nginx: ssl_protocols TLSv1.2 TLSv1.3;
  9. Testar novamente — SSL Labs + curl + GSC URL Inspection
  10. Configurar alerta — monitoramento de expiração (UptimeRobot, cron, ou do provedor)
  11. Documentar — registrar data de renovação, CA, domínios cobertos no runbook
Melhores práticas: Use Let's Encrypt com renovação automática (Certbot). Ative HSTS. Configure redirect 301 direto de HTTP para HTTPS sem chain. Monitore a expiração com alerta 30 dias antes.

Exemplo — Nginx com TLS moderno

server {
    listen 443 ssl http2;
    server_name site.com www.site.com;

    ssl_certificate     /etc/letsencrypt/live/site.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/site.com/privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
    ssl_prefer_server_ciphers on;

    add_header Strict-Transport-Security "max-age=63072000" always;
}

server {
    listen 80;
    server_name site.com www.site.com;
    return 301 https://$host$request_uri;
}

FAQ

O Googlebot realmente não indexa páginas com TLS expirado?
Sim. Ele usa o mesmo trust store do Chrome. Se a conexão falha, ele não consegue baixar o HTML. O GSC mostra como "Crawled but currently not indexed" ou "Page inaccessible".
E se eu usar Cloudflare? O TLS é gerenciado por eles?
Sim, Cloudflare fornece certificado automático. O problema ocorre se o modo for "Flexible" em vez de "Full (strict)" — a conexão de origem fica HTTP. Use sempre "Full (strict)".
Quanto tempo leva para o Googlebot voltar a indexar depois da correção?
Assim que o Googlebot tentar recrawlear e encontrar a conexão funcionando, a indexação é retomada. Use URL Inspection para solicitar indexação imediata. Normalmente leva de algumas horas a 2 dias.
Qual a diferença entre SSL e TLS?
SSL (Secure Sockets Layer) é o protocolo original, obsoleto e vulnerável. TLS (Transport Layer Security) é o sucessor. Hoje todo "certificado SSL" é na verdade TLS. Você quer TLS 1.2 ou 1.3.
Preciso de certificado pago ou Let's Encrypt é suficiente?
Let's Encrypt é suficiente. O Googlebot não diferencia CA paga de gratuita. O requisito é ser emitido por uma CA confiável, e Let's Encrypt é reconhecida por todos os navegadores e crawlers.
O que é um certificado SAN e quando preciso?
SAN (Subject Alternative Name) permite que um único certificado cubra múltiplos domínios: site.com, www.site.com, blog.site.com. Sem SAN, cada domínio precisa do seu próprio certificado.

02 Robots.txt bloqueando crawlers

Crítico Crawl

Visão geral

O robots.txt é o primeiro arquivo que o Googlebot solicita ao chegar em um site. Se ele bloqueia diretórios inteiros, URLs específicas ou arquivos JS e CSS necessários para renderização, o crawler não consegue acessar nem indexar o conteúdo.

Causa raiz

Configuração incorreta: Disallow: / (bloqueia tudo), Disallow: /wp-admin/, ou bloqueio acidental de assets .js e .css que o Googlebot precisa para renderizar.

Diagnóstico

curl -A "Googlebot" https://seu-site.com/robots.txt
# GSC > Config > Robots.txt mostra o conteúdo e erros de parse

POP

  1. Acessar /robots.txt e confirmar o conteúdo
  2. Verificar se Disallow: / está presente (só deve estar em staging)
  3. Verificar se assets .js e .css estão acessíveis (não bloqueados)
  4. Usar o tester do GSC para simular bloqueios por URL
  5. Corrigir permissões no servidor web se arquivos retornam 403
Se eu bloquear JS no robots.txt, o Googlebot ainda renderiza a página?
Não. Googlebot precisa dos arquivos JS e CSS para renderizar o DOM completo. Se estão bloqueados, ele vê apenas o HTML inicial, que em SPAs é vazio.

03 Noindex acidental

Crítico Indexação

Visão geral

A meta tag <meta name="robots" content="noindex"> ou o header X-Robots-Tag: noindex instrui o Googlebot a não incluir a página no índice. Quando colocado acidentalmente em páginas importantes (home, categorias, posts), elas somem da pesquisa.

Causa raiz

  • Plugin de SEO configurado incorretamente
  • Template que aplica noindex em todas as páginas por padrão
  • Ambiente de staging com noindex que vazou para produção
  • Tema WordPress com noindex por estar "Coming Soon"
  • Header HTTP definido por servidor web ou CDN

Diagnóstico

curl -s https://seu-site.com | grep -i "noindex\|nofollow"
curl -I https://seu-site.com | grep -i "x-robots"
# GSC > URL Inspection mostra se a página tem noindex

POP

  1. Identificar páginas afetadas via GSC (Index Coverage: "Excluded by noindex tag")
  2. Verificar source: meta tag no <head> ou header HTTP
  3. Remover o noindex de páginas que devem ser indexadas
  4. GSC URL Inspection > "Solicitar indexação" após correção
  5. Configurar alarme no GSC para detectar picos de páginas com noindex
Qual a diferença entre noindex e nofollow?
Noindex impede a indexação da página. Nofollow diz ao Googlebot para não seguir os links da página (mas ela pode ser indexada). Podem ser usados juntos: noindex, nofollow.

04 Sitemap XML com erros

Alta Crawl

Visão geral

O sitemap.xml é o principal canal de descoberta de URLs para o Googlebot. Erros de formato, URLs quebradas, redirects, ou a ausência total do sitemap fazem com que páginas importantes nunca sejam descobertas.

ProblemaConsequência
Sitemap ausenteGoogle descobre URLs apenas por links internos
URLs com redirect (3xx)Desperdiça crawl budget
URLs com erro (4xx/5xx)Crawler gasta recursos em páginas que não existem
URLs com noindexContradição: pede para indexar mas não quer que indexe
Sitemap acima de 50 MB ou 50 mil URLsGooglebot ignora linhas excedentes

POP

  1. Verificar se o sitemap existe em /sitemap.xml ou no path do robots.txt
  2. Validar formato XML (<urlset> com namespaces corretos)
  3. Confirmar que cada URL retorna 200 (sem redirects ou erros)
  4. Remover URLs com noindex, canonical externo, ou bloqueio do robots.txt
  5. Submeter no GSC > Sitemaps > "Adicionar sitemap"
  6. Manter atualizado — gerar dinamicamente ou via cron
Sitemap substitui a necessidade de links internos?
Não. Sitemap ajuda na descoberta, mas links internos são essenciais para passar autoridade (PageRank) e contexto temático entre páginas.

05 Canonical mal configurado

Alta Indexação

Visão geral

A tag <link rel="canonical"> diz ao Google qual versão de uma URL deve ser tratada como a principal. Quando aponta para a página errada, para uma URL que redireciona, ou para si mesma em páginas paginadas, o Google pode indexar a versão errada ou ignorar a página.

Causa raiz

  • Canonical apontando para URL com redirect em vez da final
  • Todas as páginas apontando canonical para a home
  • Páginas paginadas (page/2, page/3) apontando canonical para page/1
  • Canonical ausente em páginas com parâmetros de tracking
  • Canonical definido via JavaScript em vez de HTML estático

POP

  1. Crawlear com Screaming Frog e exportar coluna "Canonical URL"
  2. Identificar páginas com canonical diferente da URL atual
  3. Verificar se o canonical aponta para URL 200 (não redirect nem erro)
  4. Corrigir: cada página única deve ter self-canonical
  5. Paginação: usar rel="prev" e rel="next"
O que acontece se o canonical aponta para URL 404?
O Google ignora o canonical e tenta determinar a URL canônica sozinho, geralmente escolhendo a versão com mais links internos.

06 Redirecionamentos quebrados

Alta Crawl

Visão geral

Cadeias de redirect (A para B para C), redirect loops e redirects para URLs 4xx ou 5xx desperdiçam crawl budget, atrasam a indexação e podem fazer o Googlebot desistir.

POP

  1. Crawlear e identificar redirects (Screaming Frog > Status Code 3xx)
  2. Reduzir chains a um único 301 sempre que possível
  3. Redirects temporários (302) que são permanentes devem virar 301
  4. Redirects para 4xx ou 5xx devem apontar para a URL correta
  5. Redirect loops: corrigir configuração conflitante (Nginx + app + CDN)
Quantos redirects o Googlebot segue?
Google diz que segue até 10 hops, mas o recomendado é no máximo 3. Cada redirect adicional desperdiça crawl budget e dilui autoridade.

07 Erros HTTP 4xx e 5xx em páginas indexadas

Alta Indexação

Visão geral

Páginas que antes estavam indexadas e passam a retornar 404 (removidas sem redirect) ou 5xx (erro de servidor) são retiradas do índice. Google interpreta 404 como "conteúdo removido" e 5xx como "site instável".

POP

  1. GSC > Index Coverage > "Not found (404)" — veja URLs com erro
  2. Para cada 404: se foi movido, colocar 301. Se foi removido de propósito, retornar 410 (Gone)
  3. Para 5xx: investigar causa (pico de tráfego, recurso esgotado, erro de aplicação)
  4. Implementar monitoramento de uptime com alertas para 5xx
  5. Corrigir links internos que ainda apontam para URLs 404

08 Conteúdo duplicado

Média Indexação

Visão geral

Conteúdo idêntico ou muito similar acessível em URLs diferentes (com e sem www, HTTP e HTTPS, parâmetros de tracking, filtros) faz o Google gastar crawl budget em duplicatas e dividir sinais de ranking.

Solução

  • Escolher um domínio canônico (www vs non-www) e redirecionar o outro
  • Usar canonical tag em todas as páginas
  • Parâmetros de tracking: configurar no GSC em Config > Parâmetros de URL
  • Filtros de categoria: usar canonical para a versão padrão
  • Paginação: usar rel="prev" e rel="next"

09 Thin content

Média Conteúdo

Visão geral

Páginas com pouco ou nenhum conteúdo útil (produto sem descrição, categorias com duas linhas, páginas geradas automaticamente) são penalizadas pelo Google e tendem a não ranquear.

POP

  1. Extrair word count de todas as páginas (Screaming Frog)
  2. Identificar páginas com menos de 300 palavras (blog) ou 100 palavras (produto/categoria)
  3. Para produtos: adicionar descrição única, especificações, reviews
  4. Para categorias: adicionar texto introdutório, guia de compra, FAQ
  5. Páginas sem conteúdo suficiente: noindex ou canonical para página mais rica

10 Páginas órfãs

Alta Arquitetura

Visão geral

Páginas que não recebem links internos de nenhuma outra página do site. Sem links internos, o Googlebot depende exclusivamente do sitemap para descobri-las. Mesmo assim, sem link interno, não recebem autoridade (PageRank) e tendem a não ranquear.

POP

  1. Screaming Frog > crawl completo > exportar "Inlinks"
  2. Identificar páginas com 0 ou 1 link interno
  3. Criar links contextuais a partir de páginas relevantes
  4. Para SPAs: garantir que o router cria <a> no HTML, não onClick em divs
  5. Auditar periodicamente com crawl diff

11 Canibalização de palavras-chave

Média Conteúdo

Visão geral

Quando duas ou mais páginas do mesmo site competem pela mesma palavra-chave, o Google precisa escolher qual mostrar. Isso divide cliques, dilui autoridade, e nenhuma ranqueia tão bem quanto ranquearia sozinha.

Diagnóstico

  • GSC > Search Results: veja quais URLs aparecem para a mesma query
  • Se duas URLs suas aparecem para a mesma keyword, há canibalização
  • Screaming Frog + export de titles + H1 ajuda a identificar sobreposição

Solução

  • Fundir páginas concorrentes em uma única, mais completa, com redirect 301
  • Diferenciar o foco de cada página (keyword diferente, intent diferente)
  • Usar canonical para a versão principal se a fusão não for possível
  • Internal link: direcionar todos os links para a página que você quer ranquear

12 Title e meta description ausentes ou duplicadas

Média On-page

Visão geral

O <title> e a <meta name="description"> são os primeiros sinais que o Google usa para entender o assunto da página. Ausentes, duplicados ou gerados automaticamente, o Google escolhe o que mostrar no SERP, muitas vezes pior do que você escreveria.

POP

  1. Screaming Frog > exportar Title 1 e Meta Desc 1 para páginas ausentes ou duplicadas
  2. Cada title: único, descritivo, abaixo de 60 caracteres, com keyword principal no início
  3. Cada meta description: única, abaixo de 160 caracteres, com call to action
  4. Corrigir no CMS ou template
  5. Em SPAs: garantir que title e description estejam no SSR, não apenas via JS

13 LCP alto (Largest Contentful Paint)

Crítico Performance

Visão geral

LCP mede o tempo que o maior elemento visível da página leva para renderizar. Limite do Google: 2,5 segundos. Acima disso, a página é considerada lenta e penalizada no ranking. Em SPAs, o LCP é particularmente problemático porque o conteúdo principal só existe após o JavaScript executar.

Causas comuns

  • Imagem de hero sem compressão e sem fetchpriority="high"
  • Fonte customizada com font-display: swap lento
  • SPA sem SSR: o maior elemento é o spinner de loading
  • Render-blocking resources carregando antes do conteúdo
  • Server response time (TTFB) acima de 500 ms

POP

  1. Identificar o elemento LCP (PageSpeed Insights > Diagnostics)
  2. Se for imagem: comprimir (WebP/AVIF), fetchpriority="high", loading="eager"
  3. Se for texto: ativar font-display: swap, preconnect para Google Fonts
  4. Se TTFB alto: otimizar servidor (cache, CDN, PHP, banco)
  5. Se SPA: implementar SSR para HTML conter o conteúdo principal
  6. Usar <link rel="preload"> para o recurso LCP

14 INP ruim (Interaction to Next Paint)

Performance CWV

Visão geral

INP mede a latência de todas as interações do usuário (cliques, toques, teclado) e reporta a pior observada. Substituiu o FID como Core Web Vital em março de 2024. Limite: abaixo de 200 ms. Em SPAs, cada navegação entre rotas pode disparar INP alto se o bundle JS for grande.

Solução

  • Code splitting: carregar apenas o JS necessário para cada rota
  • Lazy load de componentes abaixo da dobra
  • Evitar long tasks acima de 50 ms
  • Em SPAs: pré-renderizar rotas via SSR
  • Debounce event handlers pesados (scroll, resize, input)

15 CLS alto (Cumulative Layout Shift)

Performance UX

Visão geral

CLS mede mudanças no layout após o carregamento inicial. Causado por elementos que "pulam" depois que outros carregam: imagens sem dimensões, anúncios, fonts, embeds, ou em SPAs, componentes que renderizam em etapas. Limite do Google: abaixo de 0,1.

POP

  1. Identificar elementos que causam shift (Lighthouse > "Avoid large layout shifts")
  2. Toda imagem precisa de width e height (ou aspect-ratio CSS)
  3. Reservar espaço para anúncios e embeds com placeholder de dimensões fixas
  4. Fonts: usar font-display: optional ou swap com size-adjust
  5. Em SPAs: evitar componentes que renderizam em fases sem altura definida
  6. Injetar CSS crítico inline para layout inicial estável

16 JavaScript bloqueante

Performance Crawl

Visão geral

Arquivos JS carregados no <head> sem defer ou async bloqueiam o parser do HTML. O navegador e o Googlebot param de ler o HTML, baixam e executam o JS, e só depois continuam. Isso atrasa LCP e TTFB percebido.

POP

  1. Chrome DevTools > Performance > veja "Long tasks" e "Main thread blocking"
  2. Identificar scripts no <head> sem defer ou async
  3. Adicionar defer para scripts não críticos
  4. Scripts críticos: inline minimizado no <head>
  5. Code splitting: bundles carregados apenas quando necessários

17 Imagens não otimizadas

Média Performance

Visão geral

Imagens representam mais de 50% do peso médio de uma página. Sem compressão, sem WebP/AVIF, sem dimensões, sem lazy loading e sem alt text, cada erro impacta LCP, CLS, acessibilidade e indexação no Google Imagens.

POP

  1. Crawlear e listar todas as imagens (Screaming Frog > Images)
  2. Identificar imagens acima de 100 KB, PNGs sem necessidade, sem alt text
  3. Converter para WebP ou AVIF com compressão inteligente (80-85% qualidade)
  4. Adicionar loading="lazy" em imagens abaixo da dobra
  5. Adicionar width e height em todas as imagens
  6. Adicionar alt descritivo com keyword natural
  7. Implementar srcset para servir imagens no tamanho correto do viewport
  8. Configurar cache no CDN com long TTL

18 Schema markup incorreto

Alta Schema

Visão geral

Dados estruturados ajudam o Google a entender o conteúdo e gerar rich snippets. Schema com erros não gera rich result e, em alguns casos, pode causar ação manual (spam de schema).

Problemas comuns

  • Tipo de schema errado para o conteúdo
  • Propriedades obrigatórias ausentes (ex: name em Product)
  • Valores em formato errado (preço como string em vez de número)
  • Schema injetado via JS que o Googlebot não vê
  • Review schema com nota 5.0 para tudo
  • FAQPAGE com perguntas que são apenas keywords

POP

  1. Validar todas as páginas com schema no Rich Results Test (Google)
  2. Corrigir erros: propriedades ausentes, tipos incorretos, formatos
  3. Schema deve estar no HTML inicial (server-rendered), nunca apenas via JS
  4. Usar JSON-LD (recomendado pelo Google), não microdata ou RDFa
  5. GSC > Rich Results > veja erros diminuírem e itens válidos aumentarem

19 JSON-LD inválido

Média Schema

Visão geral

JSON-LD inválido (mal formatado, trailing commas, aspas simples, @context ausente) é ignorado pelo Google sem aviso. A página simplesmente não gera rich result. Diferente de HTML, o parser de JSON é estrito.

Diagnóstico

curl -s https://seu-site.com | python3 -c "
import sys, json, re
html = sys.stdin.read()
for match in re.finditer(
  r'',
  html, re.DOTALL):
    try:
        data = json.loads(match.group(1))
        print(json.dumps(data, indent=2))
    except json.JSONDecodeError as e:
        print(f'ERRO: {e}')
"

20 Mobile usability

Crítico Mobile

Visão geral

O Google usa indexação mobile-first desde 2021. Se a versão mobile do site tem problemas de usabilidade, ela é usada como base para o ranking. Erros como conteúdo maior que a tela, touch elements muito próximos, fontes minúsculas e Flash não suportado penalizam o site.

POP

  1. GSC > Mobile Usability > veja os erros reportados
  2. Corrigir viewport: <meta name="viewport" content="width=device-width, initial-scale=1">
  3. Aumentar font-size para no mínimo 16px
  4. Aumentar espaçamento entre touch targets (mínimo 48x48 px)
  5. Garantir que o conteúdo não exceda a largura do viewport
  6. Testar manualmente em device mode no Chrome DevTools

21 Viewport configurado incorretamente

Mobile UX

Visão geral

A meta tag viewport diz ao navegador como renderizar a página no dispositivo. Sem ela, o navegador assume desktop width e aplica zoom: o site parece "zoomado", conteúdo cortado, e o Google marca como "Not mobile-friendly".

Solução

<meta name="viewport" content="width=device-width, initial-scale=1">

Nunca usar maximum-scale=1 ou user-scalable=no (impede zoom e é considerado anti-acessibilidade).

22 Single Page Application sem SSR ou SSG

Crítico JavaScript

Visão geral

SPAs (React, Vue, Angular) sem SSR ou SSG entregam ao crawler um HTML praticamente vazio: <div id="root"></div>. Todo o conteúdo da página (textos, headings, links, metadados) só existe após o JavaScript ser baixado, parseado e executado. O Googlebot vê uma página em branco.

Este é o erro mais comum e mais grave em sites JavaScript modernos. Ele afeta não só o Google, mas todos os crawlers de IA (ClaudeBot, GPTBot, PerplexityBot) que não executam JavaScript.

Causa raiz

Frameworks JavaScript (React CRA, Vue CLI, Angular CLI) geram por padrão uma aplicação CSR. O servidor envia um HTML esqueleto com um único <div> e um bundle JS. Mas crawlers se comportam de forma diferente:

  • Googlebot: renderiza JS em segunda wave (fila). Pode levar dias ou semanas
  • Bing, Yahoo: geralmente não executam JS
  • AI Crawlers (GPTBot, ClaudeBot, PerplexityBot): não executam JS. Indexam zero
  • Social crawlers (Facebook, Twitter, LinkedIn, Slack): não executam JS

Impacto no SEO

  • Páginas não indexadas ou "Crawled - currently not indexed"
  • Zero visibilidade em AI search (ChatGPT, Perplexity, AI Overviews)
  • Rich snippets ausentes (schema injetado via JS não é lido)
  • Social previews quebrados
  • Crawl budget exaurido com rendering sem garantia

Diagnóstico

1. Teste com JavaScript desabilitado

Chrome DevTools > Settings > Debugger > "Disable JavaScript". Recarregue. O que você vê é o que a maioria dos crawlers vê.

2. curl com user-agent Googlebot

curl -s -A "Mozilla/5.0 (compatible; Googlebot/2.1; \
  +http://www.google.com/bot.html)" \
  https://seu-site.com | head -50
# Se for <div id="root"></div> + script, é CSR puro

3. GSC > URL Inspection > "Test Live URL" > "Rendered HTML"

Compare o rendered HTML com o source original. Se são drasticamente diferentes, você depende de JS rendering.

4. Tamanho do HTML inicial

curl -s -A "Googlebot" https://seu-site.com | wc -c
# HTML abaixo de 1 KB = CSR shell vazio
# Acima de 10 KB com texto = SSR ou SSG

Solução

AbordagemFuncionaTrade-off
SSR (Next.js, Nuxt, Angular Universal)Server renderiza HTML completo a cada requestServidor Node.js necessário; TTFB maior
SSG (Astro, Next.js export, Hugo)HTML gerado em build time, servido estáticoRebuild para atualizar conteúdo
ISR (Next.js)SSG + regeneração sob demandaInfra Next.js; cache invalidation
Prerendering (Rendertron, Prerender.io)Headless browser gera HTML para crawlersWorkaround; Google não recomenda como solução definitiva

POP — SSR ou SSG

  1. Diagnosticar: curl + GSC para confirmar CSR puro
  2. Escolher: SSR se conteúdo dinâmico, SSG se estático, ISR se frequência média
  3. Next.js SSR: migrar pages para getServerSideProps ou App Router
  4. Next.js SSG: usar getStaticProps com revalidate para ISR
  5. Astro SSG: arquivos .astro geram HTML estático por padrão
  6. Garantir meta tags, canonical, OG tags e JSON-LD no HTML inicial
  7. Garantir que links usam <a> com href
  8. Testar: curl com Googlebot deve mostrar texto, headings e links
  9. GSC URL Inspection: solicitar indexação das páginas principais
  10. Testar em AI crawlers: DataJelly Visibility Test
Atenção: Dynamic rendering é um workaround, não uma solução definitiva. Google recomenda SSR ou SSG. Se usar dynamic rendering, o conteúdo para crawlers deve ser estritamente equivalente ao que usuários veem, senão é cloaking.

Exemplo — Next.js SSR

export async function getServerSideProps({ params }) {
  const res = await fetch(`https://api.site.com/produtos/${params.id}`);
  const produto = await res.json();
  return { props: { produto } };
}

export default function Produto({ produto }) {
  return (
    <>
      <Head>
        <title>{produto.nome} | Loja</title>
        <meta name="description" content={produto.descricao} />
      </Head>
      <h1>{produto.nome}</h1>
      <p>{produto.descricao}</p>
    </>
  );
}

Exemplo — Astro SSG

---
const { id } = Astro.params;
const res = await fetch(`https://api.site.com/produtos/${id}`);
const produto = await res.json();
---
<html>
  <head>
    <title>{produto.nome} | Loja</title>
    <meta name="description" content={produto.descricao} />
  </head>
  <body>
    <h1>{produto.nome}</h1>
    <p>{produto.descricao}</p>
  </body>
</html>

FAQ

Meu site é React puro (CRA). Preciso reescrever tudo?
Depende. Se tem poucas páginas estáticas, use prerendering como ponte. Se tem muitas páginas dinâmicas, migre para Next.js incrementalmente. O custo de não fazer: zero indexação para crawlers que não executam JS.
O Googlebot executa JavaScript ou não?
Sim, mas em segunda wave (fila de rendering) que pode levar dias. AI crawlers e redes sociais não executam JS. Se seu conteúdo depende de JS, você perde todos esses canais.
Sitemap resolve o problema de SPA?
Não. Sitemap ajuda a descobrir URLs, mas não muda o que o crawler vê quando chega lá. Se o HTML é vazio, sitemap não adianta. O problema é de rendering, não de descoberta.
Dynamic rendering é cloaking?
Não, desde que o conteúdo servido para crawlers e usuários seja equivalente. Cloaking seria servir algo diferente (encher de keywords para crawler).
Next.js com SSG é suficiente para SEO?
Sim, SSG é a melhor opção para SEO se o conteúdo não muda por usuário. HTML gerado em build time, crawler recebe página completa. Única limitação: conteúdo que muda com frequência exige rebuild ou ISR.
Como testar se meu site SPA é indexável?
1) curl -A "Googlebot" https://site.com | wc -c — se HTML abaixo de 1 KB, problema. 2) GSC URL Inspection > Rendered HTML. 3) Desabilite JS no Chrome e recarregue. 4) DataJelly Visibility Test.

23 Two-wave indexing — a fila de rendering do Google

JavaScript Indexação

Visão geral

O Googlebot não processa JavaScript em um único passo. Usa um processo de duas ondas (two-wave indexing). A Wave 1 é rápida: baixa o HTML, indexa o conteúdo estático. Wave 2 é diferida: a página entra numa fila de rendering que pode levar de segundos a semanas.

Impacto

  • Conteúdo novo em CSR: leva dias para indexar (horas em SSR)
  • Atualizações de preço, título, descrição: propagação lenta
  • Meta tags, canonical, hreflang, schema: ausentes na Wave 1 se dependem de JS
  • Crawl budget: páginas na fila consomem recursos sem garantia

Solução

Única solução definitiva: conteúdo crítico no HTML inicial (SSR ou SSG). Servir HTML completo elimina a dependência da Wave 2.

24 Hash routing — URLs com # não são indexáveis

Crítico SPA

Visão geral

URLs como site.com/#/produto/123 usam hash fragments. Tudo depois de # nunca é enviado ao servidor e só existe no browser. Googlebot ignora o fragmento e trata todas como site.com/. Cada "página" da SPA fica invisível.

Solução

Migrar para History API (HTML5 pushState): site.com/produto/123. React Router, Vue Router e Angular Router já usam History API por padrão. Verifique se o servidor serve index.html para todas as rotas.

# Nginx — servir app shell para todas as rotas SPA
location / {
    try_files $uri $uri/ /index.html;
}

25 Soft 404 em SPAs

SPA Indexação

Visão geral

Em SPAs CSR, o servidor sempre retorna 200 (o mesmo index.html) para qualquer URL. Quando o JavaScript decide que a rota não existe, renderiza um componente "404", mas o HTTP response já foi 200. Isso é um soft 404: o Google indexa a página como válida, mas o conteúdo diz "não encontrado".

Impacto

  • Google indexa páginas de erro como conteúdo válido
  • Crawl budget desperdiçado em URLs que não existem
  • Usuário cai em página de erro vinda do Google

Solução

Solução real: SSR. O servidor verifica se a rota existe e retorna 404 real. Para CSR, as mitigações possíveis:

  • Redirecionar o componente 404 para URL que o servidor retorna 404
  • Adicionar <meta name="robots" content="noindex"> via JS na página de erro
  • Não incluir soft 404s no sitemap

26 Lazy loading que impede o crawl

Performance Crawl

Visão geral

Lazy loading é ótimo para performance, mas problemático se o conteúdo principal é carregado com base em scroll ou clique. Googlebot não simula scroll e não clica em "Carregar mais".

Regras para lazy loading SEO-safe

  • Native loading="lazy" em imagens: seguro
  • Intersection Observer para imagens: seguro
  • Conteúdo carregado on scroll: não seguro
  • Conteúdo atrás de clique ("Ver mais"): não seguro sem URL separada
  • Infinite scroll: não seguro sem fallback de paginação com <a>

27 Infinite scroll sem paginação crawleável

SPA Crawl

Visão geral

Infinite scroll carrega conteúdo conforme o usuário rola. Googlebot não rola. Se o único jeito de acessar o conteúdo é rolando, o crawler só vê os primeiros itens.

Solução

Implementar paginação tradicional em paralelo ao infinite scroll:

  • Cada "página" com URL única: /blog/page/2, /blog/page/3
  • Links <a> de paginação no HTML
  • <link rel="next"> e <link rel="prev"> no <head>
  • Usuários veem infinite scroll; crawlers veem os links <a>

28 JavaScript bloqueado no robots.txt

Crítico Crawl

Visão geral

Se o robots.txt bloqueia diretórios de assets JS ou CSS, o Googlebot não consegue baixar os arquivos necessários para renderizar a página. O resultado: página quebrada, sem conteúdo, considerada de baixa qualidade ou não indexada.

Diagnóstico

curl -A "Googlebot" https://seu-site.com/robots.txt
# Procure por Disallow contendo .js ou /assets/js/
# GSC > URL Inspection > veja se assets aparecem como "Blocked"

Solução

Remover Disallow de arquivos .js e .css no robots.txt.

User-agent: *
Disallow: /admin/
Disallow: /private/
# NUNCA: Disallow: /assets/js/
# NUNCA: Disallow: *.js

29 Problemas de hydration em SPAs

SPA Performance

Visão geral

Hydration é o processo em que o JavaScript "assume o controle" do HTML servido pelo SSR. Quando o JS hidrata, espera que o DOM gerado pelo servidor seja idêntico ao que ele geraria. Diferenças causam hydration mismatch: o conteúdo pisca, desaparece, muda de posição, confundindo crawlers e prejudicando CLS.

Problemas comuns

  • Conteúdo gerado no server vs cliente diferente (data atual, preço com desconto)
  • CSS-in-JS que gera classes diferentes no server vs client
  • Componentes que dependem de window, document, localStorage
  • Third-party scripts que modificam o DOM durante hydration

Solução

  • Para conteúdo dinâmico, renderizar do lado do cliente após hydration com useEffect
  • Garantir estado inicial idêntico entre server e client
  • Next.js: dynamic(() => import(...), { ssr: false }) para componentes client-only
  • Islands architecture (Astro): hidrata apenas componentes interativos
